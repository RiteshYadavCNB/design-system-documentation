Why Atomic Design?

Atomic Design ensures every UI decision is traceable back to a decision in your token system. When a color changes, it changes everywhere — in Figma, in code, in the product — because it was defined once and referenced everywhere.

Core Philosophy

The three principles that govern every design decision in this system.

1. Single Source of Truth ▶
Every value — a color, a spacing unit, a font size — must be defined once as a token and referenced everywhere else. The moment a designer types #3B82F6 directly in a fill, that value is no longer part of the system. It is a defect. The design system is only as strong as the strictness with which it is followed.
2. Composability Over Duplication ▶
Before creating any new component, ask: can this be built from existing atoms and molecules? A new component should be a new composition, not new raw material. Duplication creates drift — two "primary buttons" that look 2px apart, two "cards" with different shadow values, because someone created one from scratch instead of using the existing one.
3. Design → Token → Code ▶
Every visual property flows in one direction: Design Decision → Named Token → Code Variable. A designer decides "our primary button background is the brand primary color." That decision becomes a Figma Variable (color/brand/primary) which maps to a code token (theme.color.brand.primary). Never the reverse — code should never define a visual value that isn't in the design system.

What This System Is Not

✓ This system IS
  • A living document that evolves
  • A shared language between design and dev
  • A constraint that speeds up decisions
  • A multiplier on individual quality
  • The source developers pull from
✕ This system is NOT
  • A rigid set of rules that blocks creativity
  • A finished product — it will always change
  • A library of every possible UI pattern
  • The responsibility of one person
  • Optional for new components

⚛ Foundation Layer

Design Tokens

Tokens are the DNA of the design system. Everything is a token, or is built from tokens. They bridge Figma Variables and code.

Three-Tier Token Architecture

// Tier 1: Primitives — raw values, never used directly in designs
color/red/500 = #EF4444
color/blue/600 = #2563EB
size/4 = 4px
size/16 = 16px

// Tier 2: Semantic — what designers apply in Figma
color/feedback/danger = {color/red/500}
color/brand/primary = {color/blue/600}
spacing/md = {size/16}

// Tier 3: Component — scoped to a specific component
button/bg/primary = {color/brand/primary}
button/padding/h = {spacing/md}
⚠️
Critical Rule: Never Skip a Tier

A button should reference button/bg/primary, not color/brand/primary directly. And button/bg/primary should reference color/brand/primary, not #2563EB. This chain is what makes theming and dark mode possible.

In Figma: Use Variable Collections

Create two collections in Figma: Primitives (hidden, raw values) and Semantic (what designers use). Enable the Semantic collection in all design files. Never expose Primitives in the color picker — they should only appear as references inside Semantic variables.

⚛ Level 1

Atoms

Atoms are the smallest possible UI units. They cannot be broken down further without losing their meaning. They are not components — they are the raw material components are made from.

🧬
What counts as an Atom?

A color is an atom. A spacing value is an atom. An icon is an atom. A text style is an atom. A single HTML element with no children that cannot be simplified further — that's an atom. A button is NOT an atom (it's a molecule).

Icon System Rules

1
Size scale: 12 / 16 / 20 / 24 / 32px only
Never resize an icon to an arbitrary value. The size scale matches the 4pt grid. Icon components should have a Size variant: xs(12) / sm(16) / md(20) / lg(24) / xl(32).
2
Naming: Icon/[category]/[name]
Example: Icon/navigation/arrow-right, Icon/action/edit, Icon/status/check-circle. Consistent naming enables search and the manifest generator to correctly map icon usage.
3
Color bound to content variable
Icon fills should be bound to color/content/primary or color/content/secondary by default. Never hardcode an icon color.

🦠 Level 3

Organisms

Organisms are complex, self-contained UI sections. They compose multiple molecules and atoms into a functional unit. A Card is an organism. A Navbar is an organism. A Form is an organism.

✓ Valid Organisms
  • Navbar (logo + nav links + CTA)
  • Card (image + content + actions)
  • Modal (header + body + footer)
  • Form (multiple inputs + button)
  • Data table with pagination
  • Product listing grid
  • Sidebar navigation
✕ These are NOT Organisms
  • A single Button → Molecule
  • An Input + Label → Molecule
  • Two buttons side by side → Molecule group
  • A full page layout → Template
  • A color token → Atom

🔗
Organism Rule: Use Instances, Not Copies

When composing an organism, always use component instances of molecules and atoms — never detach and copy. If you detach, changes to the molecule no longer propagate to the organism. The entire value of the system collapses.

📄 Level 5

Pages

Pages are templates populated with real content. They are the final design deliverable — what stakeholders review, what developers build from, and what gets tested by QA.

✓
Pages are the only place for real content

Real product copy, actual images, real pricing, actual user data scenarios — these only appear in Pages, never in Templates or below. This protects the system's reusability.

Page Definition of Done

All content is real (not placeholder)
All three breakpoints are designed (mobile, tablet, desktop)
All interactive states are shown (empty state, loading, error, success)
Dev notes are added to key interactions using Figma annotations
Stakeholder review is complete and sign-off received
Dev Mode link is shared with engineering

Figma Rules

These rules are non-negotiable. Every designer on the team must follow them. They are what make MCP integration, dev handoff, and the manifest generator work correctly.

The 10 Figma Commandments

1
Name every frame, every time
No frame named "Frame 47" or "Group 3" leaves the Sandbox page. Every component frame must have a deliberate, meaningful name following the naming convention. Default Figma names are a red flag.
2
Every fill must be a Variable
No hardcoded hex values in fills or strokes. Every color applied in a design must be a bound Figma Variable from the Semantic collection. The DS Bridge plugin will flag violations.
3
Every spacing value must be a Variable
Padding, gap, and margin values in Auto Layout must be bound to spacing variables. Never type "16" manually in the padding field — select spacing/4.
4
Use Auto Layout for everything
No absolute positioning within components. All internal layout must use Auto Layout with proper horizontal and vertical constraints. This is what makes components resize correctly and what dev gets clean flex/grid output from MCP.
5
Components before raw shapes
Before drawing any shape, check if it already exists as a component. Use the instance, not a copy. If a variant doesn't exist, add it to the component — don't detach and modify.
6
Define all states as Component Properties
Hover, pressed, disabled, error, loading — all states must be variants in the Component Set, not separate frames elsewhere in the file. This is what makes prototype interaction work and what Cursor reads for state-aware code generation.
7
Apply Text Styles, not manual font settings
Never manually set font size, weight, or line height on a text layer. Always apply a named Text Style. If the right style doesn't exist, add it to the Foundations page — don't improvise.
8
Use Sections to organize the canvas
Every page must be organized with named Sections. Components inside a Section inherit the section name in the layer hierarchy. This dramatically improves MCP output quality because the agent understands context from the section name.
9
No design in the Sandbox after it's been approved
The Sandbox page is for exploration only. Once a component is approved, it moves to its correct level page. Working directly in the Sandbox and sharing those links to devs is not allowed.
10
Run DS Bridge before handoff
Before any component is marked ready for development, run the DS Bridge plugin's Audit command. Zero hardcoded values required. Fill in the code path. Export the manifest entry. Then hand off.

Figma Variables Setup

This is the step-by-step guide for setting up Figma Variables correctly. Follow this exact sequence. Getting the structure right here makes everything downstream — dark mode, MCP, tokens export — work automatically.

Step-by-Step Setup

1
Create the Primitives Collection
Open Variables panel → New Collection → Name it "Primitives". Add all raw color values (full palette: every color at 50–950 scale). Add size values (4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96). Do not hide this collection yet.
2
Create the Semantic Collection with Modes
New Collection → Name it "Semantic". Add two modes: "Light" and "Dark". For each semantic token, set Light value to a Primitive alias and Dark value to a different Primitive alias.
3
Suppress Primitives from Picker
In the Primitives collection, enable "Hide from publishing". This removes it from the variable picker dropdown — designers will only see Semantic. This prevents accidental primitive usage.
4
Apply Semantic Variables to all existing components
Go through every component on Atoms, Molecules, Organisms pages. For each fill: click the color → switch from hex to Variable → select the correct Semantic token. Run DS Bridge Audit to verify coverage.
⚠️
Migration is the hardest part

Existing components with hardcoded colors must be migrated. This is manual work. Prioritize: Atoms first, then Molecules, then Organisms. Use the DS Bridge Audit to track progress — the hardcoded rate dropping toward 0% is your north star metric.

Dev Handoff

The handoff process is a structured workflow, not a "drop the Figma link in Slack" moment. Every handoff must include the right context so developers can build without guessing.

Handoff Package (per component)

Figma Dev Mode link — not the editor link, the Dev Mode link
DS Bridge manifest entry — JSON with code path, tokens, variants
Interaction notes — what happens on hover, press, focus, error
Edge cases documented — empty state, long text, overflow behavior
All breakpoints provided — mobile, tablet, desktop frames visible
Assets exported — icons as SVG, images at 2x, in the Figma export panel
🤖
For AI-assisted development (Cursor / Claude Code)

When handing off to a developer using Cursor or Claude Code, provide: the Figma node link, the manifest entry for this component, and the .cursorrules file path in the repo. The agent will use all three together. Never provide just the Figma link — it gets raw hex values without the manifest context.
